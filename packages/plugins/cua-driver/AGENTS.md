# CUA Driver Plugin

This package owns the OneWorks-facing lifecycle around the native Cua Driver.

- `bin/cua-driver.cjs` and `bin/runtime.cjs`: installation lookup, daemon startup, stale IPC recovery, and macOS permission preflight. These prerequisites must stay invisible to normal user tasks.
- `bin/cursor-runtime.cjs`: session-scoped pointer identity, color selection, starting position and motion, safe SVG materialization, and the cross-process “motion/style/start + pointer action” lock. Reuse `@oneworks/cursor` for the visual SVG design; keep defaults, automatic/session colors, Agent configuration, lifecycle and motion inside this plugin. Apply motion through the live MCP session before its first pointer action; daemon preflight cannot configure a future multi-cursor session. Every click-family action must pass through this transaction; never expose arbitrary image paths or raw style mutation tools to agents.
- `bin/file-lock.cjs`: identity-preserving cross-process file locks shared by app-resource scheduling and the global pointer transaction. Resource identity must never be compressed into a finite port bucket; use the full hashed lock path so unrelated apps cannot block one another. The lock heartbeat owns stale-process recovery.
- `bin/mcp-proxy.cjs`: the OneWorks safety boundary around upstream MCP tools. Keep its explicit external allowlist narrow; raw pointer movement, runtime-config mutation, cursor disabling, trajectory replay, and child-session recording must not be exposed. `set_session_cursor_color` and `set_session_cursor_start` are local state only and must not mutate the daemon until a serialized pointer action runs. An `execute_workflow` without `cursor_start` resets its next pointer action to the main-display center.
- `bin/workflow-runtime.cjs`: session-local workflow execution, cross-process app-resource scheduling, semantic target resolution, bounded waits/checkpoints, compact results, and progressive step lookup. `execute_workflows` may run independent apps concurrently; workflows sharing any bundle id stay serial even across MCP sessions, and pointer actions still use the cross-process cursor transaction. When no window is declared, select the app-title match or largest visible current-Space window rather than trusting upstream order. Upstream may repeat the same physical AX node under duplicate roots; collapse matches only when semantic fields and a complete finite frame are identical, while preserving ambiguity for same-name controls at different positions or without reliable geometry. The only permitted internal runtime mutation is pinning upstream `capture_mode` to `ax` once per MCP session so semantic workflows never depend on unstable screenshot/SOM parsing. Keep durable cross-session orchestration outside this plugin.
- `mcp/cua-driver.json`: projects the upstream native CUA tools into agent sessions.
- `bin/evidence-mcp.cjs`: internal diagnostic utility for legacy trajectory artifacts. It is intentionally not projected from `mcp/`; child sessions must never receive recording-finalization tools.
- `skills/cua-driver/`: intent routing, workflow-first selection, and action semantics only. Do not move daemon commands, binary discovery, permission preflight, workflow execution guarantees, or media repair into the skill.
- `server/src/`: manager/workspace status and launcher integration; it is not the MCP transport owner.

Run `pnpm -C packages/plugins/cua-driver test`, `pnpm -C packages/plugins/cua-driver typecheck`, and the repository typecheck after changing plugin assets or runtime contracts.
