# Client Plugins

This directory hosts the workspace client plugin runtime.

- `plugin-manifest.ts`: narrow frontend copy of the `/api/plugins` response and contribution contracts until shared package types are available.
- `plugin-i18n.ts`: host-provided plugin i18n helper. Client plugins should read language, localized text selection, contribution title/description resolution, and language-change subscription from `ctx.i18n` instead of guessing browser or URL locale themselves.
- `plugin-registry.ts`: scoped in-memory registry for commands, slots, routes, views, plugin-to-plugin extension points, launcher providers, diagnostics, and cleanup.
- `plugin-runtime.ts`: activation context, host React singleton exposure, host notification queue bridge, dynamic client entry import, scoped API helper, and hot reload plumbing.
- `PluginProvider.tsx`: React provider that fetches `/api/plugins`, activates client entries, and exposes registry snapshots.
- `PluginProvider.tsx` also owns the plugin watch websocket subscription. `plugin.changed` events should refresh plugin instances and re-import the changed plugin without reloading the whole app.
- `PluginHost.tsx`: route and view mounting helpers for plugin-rendered UI.
- `plugin-host-components.tsx`: host-rendered shared components, common controls, overlay dropdowns, and overlay primitives that DOM plugin views can mount through `view.components.render(...)`.
- `plugin-slots.tsx`: hooks for built-in UI surfaces to read plugin slot contributions. Slot consumers receive localized `title` / `description` values resolved from contribution `titleI18n` / `descriptionI18n`.
- Settings subpages are a host slot: plugins contribute `settingsPages` / `settings.pages` using either `clientView` for plugin-owned rendering or `schema` plus optional `uiSchema` for a host-rendered options form. The Settings route owns navigation, route identity, schema controls, and option persistence; custom pages mount through `PluginViewHost` with the `settings` surface and use `view.ui.SettingsSection` / `SettingsRow` for native layout.
- Session sidebar grouping is a host slot: plugins contribute `sessionGroups` / `sessions.groups` with declarative match rules and optional header actions; sidebar rendering and action styling stay host-owned.
- Chat tool-use presentation is a host slot: plugins contribute `toolUsePresentations` / `chat.toolUse.presentations` with scoped matching and declarative title, icon, target, field, and result metadata. Object arrays use host `records`, short primitive arrays use `chips`, and large protocol results use declared result fields; matching and localization live in `plugin-tool-use.ts`, while chat owns rendering. Do not add plugin-specific tool renderers to the host.

`PluginViewHost` owns the per-render `PluginViewContext`: it passes current language, i18n text resolution, theme mode, resolved theme, dark-mode state, surface (`route` / `workbench` / `drawer` / `launcher` / `settings`), plugin-to-plugin `extensions` lookup helpers, scoped `options.value` / `options.update(...)`, `runtime.endpoint` / `runtime.invokeChannel(...)` / `runtime.listEndpoints(...)`, imperative `components.render(...)`, and declarative `ui.*` React host components to each plugin view. The rendered host node also carries `data-plugin-surface` plus a `plugin-view-host--<surface>` class so platform CSS and tests can select surface modes without plugin-local DOM guesses. Add new view-scoped host capabilities here rather than through the activation-only `PluginClientContext`.

Launcher and workspace plugin pages are host surfaces of the same client plugin runtime. Workspace route / workbench / drawer injection and launcher page injection should both mount plugin-owned UI through `PluginViewHost` so they receive the same `PluginViewContext`, i18n, theme, scoped options, extension lookup, runtime channel bridge, notifications, and host component access. Future launcher plugin pages should extend `PluginViewHost` / shared `view.ui.*` / `view.components.render(...)` capabilities instead of adding launcher-only injection paths, launcher-only React singletons, or plugin-local host shims.

Launcher-visible plugin business entries must be registered by the plugin, not mirrored in `LauncherRoute`. Use manifest `routes` with the `launcher` surface for plugin pages, `launcherSearchProviders` plus `ctx.commands.register(...)` for command-palette results, and `view.route.setLauncherChrome(...)` for route chrome. Launcher search results may declare their own command-list group with `groupId/groupTitle` (or `sectionId/sectionTitle` aliases); host code should only group and render those generic fields, falling back to the default plugin group when absent. The host may add generic contribution contracts or host components when a capability is missing, but it must not add per-plugin built-in commands, view modes, menu items, or plugin-specific availability checks.

The host owns common view capabilities; plugins own business content. If a launcher or workspace plugin page needs tabs, native lists, search, icons, dropdowns, overlays, copy actions, or density controls, expose or reuse a structured host component rather than asking each plugin to import AntD, restyle shared controls, or duplicate DOM/CSS. List, search, and icon behavior follows `packages/components` `admin-list-surface` plus `.oo/rules/frontend-standard/styles.md`: divider lists use 10px vertical row padding with no row gap; gap/no-divider lists use a 10px container gap with no row padding; launcher plugin pages use host `InteractionList` with `mode="launcher"` instead of plugin-local launcher CSS, and that launcher mode follows the command-shell compact list rhythm with no list-container padding; native icon/avatar slots are 18px by default, with launcher list mode owning its 20px command icon slot; plugin search uses host `SearchInput` / `ListSearchInput` instead of plugin-local affix wrappers or custom search styling. React plugin views should branch on `view.host.surface` only to select host component modes and available actions, not to fork markup or style systems.

When `view.host.surface` is `launcher`, plugin views must behave as launcher command-shell content. The launcher host owns the top search field and outer content padding; plugin content must not add a second launcher header, sticky search row, or plugin-local outer padding reset. Lists use host `InteractionList` / `ListSearchInput` standards, preferably `InteractionList` with `mode="launcher"` and its `search` configuration, so search, grouping, row spacing, icon slots, and empty states are configured through host component props. Do not hand-roll search bars, group headers, gap / divider CSS, or AntD affix wrappers inside launcher plugins. Route, workbench, and drawer surfaces may keep regular page search when that matches the workspace page UX, while still consuming host search/list controls.

Launcher route chrome is declarative. A plugin route should describe its page icon, avatar, title, and optional search title through `view.route.setLauncherChrome(...)`; launcher renders that chrome in the command search row and renders plugin breadcrumbs/actions with the shared `route-container-inline-breadcrumb` convention. Plugin views should not render duplicate launcher heroes or breadcrumb/header bars just to show the current object. If a page needs a back stack, use `view.route.setBreadcrumb(...)`, and the launcher breadcrumb back button must call that route back handler before falling back to launcher exit. If it needs command buttons, use `view.route.setActions(...)`; hide workspace-only or high-noise actions when `view.host.surface` is `launcher`.

Plugin-to-plugin extension points are not built-in host slots. The owning plugin registers `extensionPoints` in manifest or `ctx.extensionPoints.register(...)`; other plugins use `ctx.extensionPoints.onAvailable('<scope>/<point>', point => ctx.extensionPoints.contribute(...))` so contribution setup is triggered when the target point exists, regardless of activation order. The owning view reads contributions with `view.extensions.getContributions(...)` and owns rendering semantics. Use `ctx.extensionPoints.has(...)` only for read-only checks, not for one-shot contribution setup.

Plugin-to-plugin pure APIs live in the client registry via `ctx.pluginApis.register({ id, inputSchema, outputSchema, handler })` and `ctx.pluginApis.call('<scope>/<id>', input)`. Calls must stay promise-based: the promise waits for the target API to be registered and then waits for the handler result. Use this for in-client process calls between plugins instead of routing through commands or server scoped APIs when no HTTP boundary is needed.

Plugin contribution labels and plugin-owned view text must use the plugin i18n helpers. Manifest / slot contribution text is resolved through `plugin-i18n.ts`, including `zh` / `zh-CN` / `zh-Hans` fallback. React plugin views should use `view.i18n.resolveText(...)`; activation-time commands, notification text, and launcher providers should use `ctx.i18n.resolveText(...)` or `ctx.i18n.getLanguage()`. Plugin UI should publish prompts through `ctx.notifications.show(...)` so the host owns source labels, markdown rendering, action callbacks, close controls, and plugin-level muting; do not reintroduce plugin-local fixed toasts.

Common UI controls exposed through `view.ui.*` and `view.components.render(...)` should wrap host-owned components such as icons, Ant Design controls, overlay dropdown triggers, and `components/overlay` primitives. Prefer `view.ui.*` for React plugin views; keep `view.components.render(...)` as the compatibility path for DOM plugins. Do not ask plugin views to import and restyle icons, AntD, menus, trees, search rows, or overlay panels themselves when a small structured host control can preserve the app design language. Prefer `OverlayDropdown` / `overlayDropdown` for real popup behavior; use primitive overlay components only when the host surface already owns popup placement.

Public naming in this module is always `plugin`.
