# 模型路由分析报告

本报告记录 2026-07-10 的仓库历史任务分析、首次组合型审阅和后续分类任务 / reasoning 复测，为 [`model-routing.md`](./model-routing.md) 的默认路由提供证据。它不是永久排行榜：模型、费率、工具和仓库任务分布变化后，应重新抽样验证。

## 结论摘要

- **最主要的节省来自拆阶段，而不是把完整复杂任务直接交给便宜模型。** 历史根任务中至少约三分之一本来就是固定命令、状态读取、启动、烟测或交付操作；历史子任务中至少 85% 已经被收窄到 Luna / Terra 可以承担的范围。
- **Sol 应是关键判断档，不是全流程默认档。** 它负责需求歧义、未知根因、跨模块契约、高风险变更和关键最终审阅；边界确定后，应把日常实现交给 Terra，把机械执行与固定验证交给 Luna。
- **GPT-5.4 / GPT-5.5 必须保留在比较集中，但不应按版本惯性默认使用。** 当前单位 token 费率下，GPT-5.4 与 Terra 同价，GPT-5.5 与 Sol 同价；它们的主要价值是既有验证基线、兼容性、Codex cloud 或特定场景的实测优势。
- **GPT-5.4 Mini 只用于纯机械任务。** 本次多文件微基准中，Mini 比 Luna 更慢、估算 credits 更高且漏掉了真实竞态，说明低单位费率不等于低任务总成本。
- **任何单个模型都不能代替验证。** 同一任务的一次运行里，GPT-5.4、Luna、Sol 找到了真实并发竞态，GPT-5.4 Mini、GPT-5.5、Terra 没找到；型号或档位不能保证单次正确。

## 历史会话分析

### 范围和方法

分析覆盖微基准创建前本机记录的全部 602 个历史会话：288 个根会话和 314 个子会话，其中 469 个已归档、133 个当时未归档。排除 1 个没有有效用户任务的中止空会话后，共 601 个有效样本。

分类依据是任务标题和首个用户 prompt 的任务形态，不读取或保存个人内容，也不把分类当作模型实际成功率。以下数字是“从任务边界看可以先使用较低档模型”的保守候选数；高风险授权、最终判断和实际验收仍由主线程负责。

### 已归档根会话

232 个有效的已归档根任务按主要意图分布如下：

| 任务形态                         | 数量 | 路由含义                                                      |
| -------------------------------- | ---: | ------------------------------------------------------------- |
| 启动、同步、状态和交付流程       |   49 | 明确命令与验收时由 GPT-5.4 Mini / Luna 执行                   |
| 自动化探针、问候、精确回复和烟测 |   22 | GPT-5.4 Mini；多工具链用 Luna                                 |
| 功能、架构和迁移                 |   75 | Sol 定边界；Terra 实现；Luna 验证与交付                       |
| 资料调研                         |   22 | 固定 schema 的收集可用 Luna；综合判断用 Terra，关键决策用 Sol |
| 诊断和审阅                       |   22 | 已知范围用 Terra；未知根因或高风险最终审阅用 Sol              |
| 显式委派 / 子任务编排            |    9 | 对每个独立验收面重新选档，不继承父任务模型                    |
| 其他                             |   33 | 按歧义、风险、耦合和验收重新判断                              |

仅“启动 / 同步 / 状态 / 交付”和“自动化 / 烟测”两类就有 71 / 232（30.6%）个明确的 GPT-5.4 Mini / Luna 候选。这还没有计算复杂根任务内部可降档的实现、验证和 Git 阶段。

### 子会话和未归档会话

- 236 个已归档子会话中，33 个是固定验证 / 启动 / 状态任务；88 个是已收窄的实现 / 修复；36 个是边界明确的 review / audit；32 个是局部诊断。前一类适合 Mini / Luna，后三类通常 Terra-first，共 189 / 236（80.1%）个明确的较低档候选。
- 78 个当时未归档的子会话均给出了明确写入范围、只读边界或固定输出 schema，主要是局部前端打磨、批量素材处理、分组审阅和固定结构调研；常规应从 Luna / Terra 开始，遇到架构、安全或关键决策再升级 Sol。
- 合并两部分，至少 267 / 314（85.0%）个历史子会话没有必要默认使用 Sol。
- 55 个当时未归档的非基准根任务中，至少 28 个只是精确回复、MCP 探针、状态读取、分支拉取、启动或烟测。结合归档记录，至少 99 / 287（34.5%）个有效根任务可直接从 Mini / Luna 起步；复杂根任务拆阶段后，实际可降档比例会更高。

### 从历史形态得到的策略

1. 根任务先用 Sol / Terra 确定边界，不要把父模型原样继承给子任务。
2. 子任务 prompt 一旦包含允许范围、禁止范围、输入、固定输出和验收，就重新评估 Luna / Terra。
3. 固定命令、检索、抽取、批处理、烟测和标准 Git 交付优先 Mini / Luna。
4. 局部 UI、既有模式内实现、常规 review 和结构化调研优先 Terra。
5. 未知根因、跨模块契约、安全 / 数据 / 发布风险和关键最终审阅才保留 Sol。

## 六模型独立会话微基准

### 设计

六个模型各使用一个干净独立 worktree、相同 prompt 和 medium reasoning，只读完成三类五分钟内任务：

1. 从根 `AGENTS.md` 提取严格的 dev-start Fast Path。
2. 理解真实前端页面的 header 组合与最小扩展点。
3. 审阅 `scripts/dev-start` 的健康服务复用和并发端口分配，并报告可证实问题或残余风险。

硬上限为 5 分钟，4 分 30 秒要求停止探索并提交已有结果；超过 5 分钟应中止并标记“不能用”。本轮六个会话均在时限内完成，无模型被标记为不能用，结果提取后均已归档。

评分共 10 分：规则检索 2 分、前端模式理解 3 分、局部实现审阅 4 分、范围与交付纪律 1 分。主线程随后核对源码，确认存在一条隐藏竞态：同 target 并发调用都可能在锁外观察到未 ready；第二个调用拿到端口锁后不重新复用检查，可能由 `stopStaleState` 杀掉第一个刚启动且健康的服务。

### 结果

| 模型          |     耗时 | 总 tokens（缓存输入） | 估算 credits |    得分 | 找到隐藏竞态 |
| ------------- | -------: | --------------------: | -----------: | ------: | ------------ |
| GPT-5.4 Mini  | 193.1 秒 |    324,614（234,240） |         3.00 |  8 / 10 | 否           |
| GPT-5.4       | 144.5 秒 |    345,301（211,968） |        11.43 | 10 / 10 | 是           |
| GPT-5.5       |  67.6 秒 |    162,162（121,728） |         8.44 |  8 / 10 | 否           |
| GPT-5.6 Luna  |  72.7 秒 |    266,384（206,080） |         2.37 | 10 / 10 | 是           |
| GPT-5.6 Terra |  72.9 秒 |     124,041（93,184） |         2.94 |  8 / 10 | 否           |
| GPT-5.6 Sol   |  76.3 秒 |    269,955（220,416） |        10.93 | 10 / 10 | 是           |

tokens 来自 Codex 会话的最终 token 统计，包含系统上下文、工具结果和模型输出，不等于纯业务 prompt 大小。估算 credits 按当日官方输入 / 缓存输入 / 输出费率计算：`(输入 - 缓存输入) × 输入费率 + 缓存输入 × 缓存费率 + 输出 × 输出费率`。它用于同轮相对比较，不代表最终账单。

### 如何解释这轮结果

- **Luna 是这组任务的最佳性价比样本。** 它与 Sol / GPT-5.4 一样找到真实竞态，耗时约 73 秒，估算消耗最低。说明边界明确、可只读取证的多工具局部审阅可以先试 Luna，但一次成功不能外推为 Luna 适合所有诊断。
- **Terra 是高效的日常工程档，但不是关键审阅的单点保证。** 它使用最少 tokens、约 73 秒完成并正确理解规则和前端模式，却漏掉隐藏并发竞态。常规实现可交给 Terra，高风险 correctness 结论仍需测试或第二审阅。
- **Sol 得到满分，但本题没有产生相对 Luna 的质量增益。** 其估算消耗约为 Luna 的 4.6 倍，因此应保留给开放、高风险或真正需要能力上限的任务。
- **GPT-5.5 最快，但单位费率与 Sol 相同且漏掉竞态。** 它可用于 Codex cloud、既有成功基线或延迟敏感且已验证的工作流，不构成新任务的默认性价比选择。
- **GPT-5.4 得到满分，但比 Luna 慢约一倍且估算消耗约为 Luna 的 4.8 倍。** 当前样本不支持把它作为新任务默认档；只在兼容、旧行为复现或仓库实测优势时保留。
- **GPT-5.4 Mini 没有兑现最低单位费率优势。** 它探索时间最长、输出最多且漏掉竞态；因此只能接纯机械、短上下文、单命令或固定转换，不能接多文件工具审阅。

这只是每个模型一次运行，不能估计稳定成功率，也可能受缓存、工具调用策略和随机性影响。关键任务至少应采用真实测试、确定性检查或不同模型 / 不同视角的第二审阅；模型档位本身不是验收证据。

## 分类任务与 reasoning 复测

首次测试仍是一组组合型只读审阅，不能代表实际修改和交付。随后使用完全通用的临时目录补跑 36 个干净独立会话，不读取仓库代码：

- 六个模型均以 medium 完成强约束数据处理、并发 TTL cache bug 修复、无框架前端交互实现和本地 Git 分支 / 提交 / 无快进合并，共 24 个会话。
- 每个模型再选择一个符合其定位的代表任务，补跑 low 和 high，复用 medium 形成 low / medium / high 对照，共 12 个会话。
- 每个会话仍设五分钟硬上限，禁止联网和安装依赖；结果由主线程使用独立测试、静态检查或 Git 历史再次验收。
- 36 个会话全部在五分钟内完成并通过独立验收，提取结果后已归档。

Medium 横向结果：

| 模型          |  通过 | 四任务平均耗时 | 四任务总 credits | 四任务总 tokens |
| ------------- | ----: | -------------: | ---------------: | --------------: |
| GPT-5.4 Mini  | 4 / 4 |       103.4 秒 |             4.86 |         491,442 |
| GPT-5.4       | 4 / 4 |        83.2 秒 |            13.77 |         493,588 |
| GPT-5.5       | 4 / 4 |        68.8 秒 |            19.93 |         399,277 |
| GPT-5.6 Luna  | 4 / 4 |        60.9 秒 |             4.28 |         305,138 |
| GPT-5.6 Terra | 4 / 4 |        83.5 秒 |             7.96 |         354,278 |
| GPT-5.6 Sol   | 4 / 4 |        68.3 秒 |            18.87 |         356,260 |

这组边界明确且可客观验收的实际修改任务里，Luna 在 bug、前端和 Git 三类任务上同时取得最低耗时与最低 credits；Mini 只在强约束数据处理上比 Luna 少约 0.06 credits，却更慢。GPT-5.5 / Sol 虽然较快，但没有产生额外验收收益，四任务总 credits 约为 Luna 的 4.4–4.7 倍。GPT-5.4 与 Terra 平均耗时相近，但 GPT-5.4 总 credits 更高。

Reasoning 复测中，18 个 low / medium / high 样本全部通过；high 相对 medium 在六个代表任务上分别增加约 22%–163% 耗时，没有提高通过率。Low 也不保证更快或更便宜，工具调用和缓存会造成波动。由此采用：多步受约束任务默认 medium；low 只用于重复评测稳定的机械任务；high 必须由更高质量、任务歧义或错误代价触发；本轮没有证据支持为五分钟任务默认使用 xhigh / max / ultra。

复测仍是每种组合一次运行，证明的是“这些低风险任务没有必要默认使用更贵模型”，不是各模型的稳定成功率，也不能外推到开放需求、安全、未知根因和跨模块架构。

## 规则分支的独立执行验收

在规则分支提交后，另建干净 worktree，让一个独立主会话处理“前端 header action + 服务端插件刷新能力 + 验收与 Git 计划”的只读任务，并要求它按本规则真实委派。

决策层表现正确：主会话把前端既有模式审阅规划为 Terra / medium，把边界明确的服务端能力盘点规划为 Luna / medium；只创建两个取证范围互斥的子任务，主会话不重复读取插件业务代码。最终也正确识别出普通列表 GET 命中运行时缓存，最小闭环需要显式 reload endpoint，而不是只加前端按钮。

执行层没有实现成本目标：当时的 `spawn_agent` schema 不支持 model / reasoning，两个 subagent 实际都继承为 GPT-5.6 Sol / xhigh。主会话同样为 Sol / xhigh，三者共使用 2,291,082 tokens，按当日费率估算约 73.03 credits。两个子任务分别约 156 秒和 140 秒完成；主会话在等待与集成后耗时 305.4 秒，超过五分钟硬上限，因此时间控制判定失败。

这次验收说明文档可以正确指导拆分和计划档位，但不能替代工具层的模型参数与硬中断。由此将以下要求提升为硬约束：实际 model / reasoning 必须可设置并核验；不能控制档位时不得声称降档；Prompt deadline 是软约束，主线程必须在截止时主动中断或停止等待。

修正规则后进行第二次独立验收，协调器改用支持显式 model / reasoning 的独立线程通道，成功创建 GPT-5.6 Terra / medium 前端 worker 和 GPT-5.6 Luna / medium 服务端 worker。平台记录显示两者分别在约 94.2 秒和 95.6 秒完成，并在报告提取后归档；模型隔离和 worker deadline 通过。

但协调器仍使用默认 GPT-5.6 Sol / xhigh，并在元数据核验、归档和长报告生成后耗时 345.7 秒，超过五分钟总上限；它在最终报告中自报约 196 秒，与平台 `durationMs` 不符。三线程合计使用 2,331,284 tokens、估算 51.11 credits：比首次继承型运行的 73.03 credits 低约 30%，但 raw tokens 基本没有下降，协调器自身就消耗约 36.65 credits。

第二次验收说明实际 worker 降档有效，但父线程仍可能吃掉大部分节省。由此继续要求：协调器本身必须按任务难度选择模型；worker cutoff 后要为集成和最终输出留出预算；最终时长只能使用平台或外部监控记录；归档和统计不得挤占用户 deadline。

应用这些约束后进行最终独立验收：协调器显式使用 GPT-5.6 Terra / medium，前端 worker 使用 Terra / medium，服务端 worker 使用 Luna / medium；三者均从同一规则分支提交启动。业务取证继续互斥，协调器不读取插件实现，worker 只检查指定入口并限制输出条数；归档由外部验收器在最终结果生成后执行。

平台记录显示，协调器耗时 135.3 秒、前端 worker 34.0 秒、服务端 worker 50.8 秒，均低于各自 deadline；实际运行档位由外部线程记录核验，三个会话在结果提取后全部归档。三者共使用 915,883 tokens、估算 15.04 credits。相对首次全部继承 Sol / xhigh 的 2,291,082 tokens、73.03 credits，raw tokens 下降约 60%，credits 下降约 79%。

最终验收通过的条件不是单纯“换成便宜模型”，而是四项同时成立：worker model / reasoning 可显式设置；协调器本身使用最低充分档位；取证范围和输出被收窄；外部平台时间与归档职责不占用协调器的 final reserve。协调器自己的接口看不到 worker 实际档位时按规则报告 unknown，外部验收器再用运行记录完成核验，没有用计划值或自报值代替。

## 建议的持续评测方式

- 保留 3–6 个五分钟内的代表任务：机械执行、既有模式实现、局部诊断、结构化研究、前端视觉判断和 Git 交付。
- 每个候选模型至少重复 3 次，再比较成功率、P50 / P95 耗时、总 credits、越界率和返工次数；不要只比较单次 tokens。
- 模型或费率变化、工具链重大升级，或某类任务连续两次失败时重跑相应样本。
- 超过任务时限立即终止并计为失败；低价模型的反复重试必须计入总成本。
- 只有当新模型在同一任务集上达到相同或更高验收通过率时，才按总成本和延迟替换已有可靠路径。

费率和官方相对能力来源见 [`model-routing-models.md`](./model-routing-models.md#信息来源与维护)。
