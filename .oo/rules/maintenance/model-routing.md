# 开发任务模型路由

本文定义开发任务如何按能力、速度和消耗选择模型。目标不是让每个任务都使用最强模型，而是在可靠完成、可验证和风险可控的前提下选择最低充分能力，并把一个复杂任务的不同阶段交给不同模型。

模型和费率会变化。每次实际分配前先查线程工具 schema；当前工具不支持本文列出的模型或 reasoning 时，选择能力定位最接近的可用项，不猜测模型名或参数。

## 路由必须能够实际执行

模型路由分为“计划档位”和“实际运行档位”。只有工具 schema 能显式设置并在运行记录中核验 model / reasoning 时，才能声称完成了降档和成本优化。

- `spawn_agent` 等委派工具如果没有 model / reasoning 参数，通常会继承父线程设置。此时可以按独立验收面拆上下文，但不能把计划中的 Luna / Terra 写成实际运行模型，也不能据此声称节省 credits。
- 以节省消耗为目标时，不要创建无法控制实际档位的继承型 subagent。用户已明确要求创建独立会话，且线程工具支持 model / reasoning 时，可用该工具按档位创建；否则由当前线程完成，或明确报告平台限制。
- 分配记录必须同时写“计划模型 / reasoning”和“实际模型 / reasoning”；实际值无法读取时写 unknown，不用计划值代替。
- Prompt 中的时间或 token 停止条件只是行为要求，不是平台硬限制。主线程必须记录 `startedAt`、deadline 和最后状态；到 deadline 后立即使用可用的 interrupt / cancel 能力，或停止等待并把该任务记为超时。不能因为 subagent 仍在思考而自动延长成本预算。
- 如果工具既不能控制模型档位，也不能在 deadline 中断，成本敏感任务默认不委派。模型更便宜但重复上下文、等待和复核成本更高时，同样由主线程直接完成。
- 协调线程也是路由对象。边界和验收已经明确的协调 / 集成任务默认从 Terra / medium 开始，不因它是父线程就沿用当前最强模型；只有协调过程本身包含开放式契约、高风险决策或未知根因时才使用 Sol。
- 总 deadline 至少拆为 worker cutoff、integration cutoff 和 final deadline。到 integration cutoff 后停止新工具调用，只使用已收集证据生成最终结果；归档、统计或补充取证不能挤占 final reserve。模型自报耗时只作参考，最终判定使用平台 `startedAt` / `completedAt` / `durationMs` 或外部监控记录。

## 先按抽象特征分类

不要先问“这是前端、PR 还是修 bug”，先从以下维度判断：

1. **歧义与开放度**：目标、正确结果和实现路径是否明确；是否存在多种合理解释。
2. **风险与可逆性**：失败能否轻易回滚；是否涉及权限、凭据、数据、发布、远端状态或用户资产。
3. **上下文耦合**：是否只触达一个已知位置；是否需要理解跨文件、跨模块、跨进程或历史决策。
4. **判断深度**：任务主要是照步骤执行、在既有模式内做工程判断，还是需要提出新方案、找未知根因和权衡取舍。
5. **验收强度**：结果能否由单一命令、测试、schema 或固定清单客观判断；还是需要综合证据、视觉判断或风险审阅。
6. **工作量形态**：是一次性短任务，还是大量重复、可并行、输入输出结构一致的任务。

据此采用三个能力档位；GPT-5.4 Mini 是受约束执行的更低成本候选，GPT-5.4 / GPT-5.5 是可用性和已验证工作流的旁路候选：

| 抽象类型 | 典型特征 | 默认模型 |
| --- | --- | --- |
| 受约束执行 | 低歧义、低风险、范围窄、步骤固定、结果可客观验证，或大量重复处理 | GPT-5.4 Mini；需要更强工具可靠性时用 GPT-5.6 Luna |
| 日常工程判断 | 中等上下文，需要理解既有模式并使用工具完成多步工作，但边界和验收已明确 | GPT-5.6 Terra |
| 开放式关键判断 | 高歧义、高风险、强耦合、未知根因、跨模块权衡或错误难以自动发现 | GPT-5.6 Sol |

以下情况直接使用或升级到 Sol：需求存在多种合理解释；修改公共契约或多个模块；处理未知根因；涉及安全、权限、凭据、数据库、不可逆状态或数据损失；需要解决 merge conflict / 历史重写；负责高风险变更的最终技术验收。

## 成本优先策略总结

Sol 是例外档位，不是开发任务的默认档位。按以下顺序选择：

1. **先判断是否命中 Sol 硬门槛**：高歧义、高风险、未知根因、跨模块契约或最终关键判断。命中任一项才优先 Sol。
2. **未命中硬门槛时先判断 Terra 是否足够**：需要理解已有代码、做日常工程判断或使用多个工具，但边界与验收清楚时用 Terra，不因任务属于“开发”或“修复”就自动使用 Sol。
3. **满足受约束执行条件时必须降档**：目标、输入、允许范围、步骤、停止条件和客观验收都明确，且失败可逆时，纯机械工作先试 GPT-5.4 Mini，需要更强工具协作时用 Luna。不要为了方便继续沿用父任务的 Terra / Sol。
4. **复杂任务按阶段降档**：Sol 把需求、根因或契约确定后，明确实现交给 Terra，机械修改、固定验证和交付操作交给 Luna；只有关键风险再次出现时才升回 Sol。

以下工作已经有较低档模型可以可靠承担，通常没有必要使用更贵模型：

| 工作形态 | 已经足够的模型 | 必要前提 |
| --- | --- | --- |
| 固定命令、状态读取、定向检索、日志收集、格式化、烟测、单一测试 | GPT-5.4 Mini low；工具链较长时 Luna low | 命令与期望结果明确，不要求判断未知根因 |
| commit、push、PR 元数据、检查状态收集、无冲突 merge | GPT-5.4 Mini 或 Luna low / medium | 用户已授权，diff 已审阅，分支保护、检查和审批满足 |
| 固定 schema 的抽取、分类、转换、批量汇总 | GPT-5.4 Mini 或 Luna low / medium | 输入输出格式和错误处理规则明确 |
| 已被父任务收窄到指定文件、接口和验收命令的子任务 | Terra medium；机械部分 Luna | 不允许自行扩大范围，遇到契约矛盾立即回报 |
| 既有设计系统内的页面、样式、响应式和局部交互 | Terra medium / high；5.6 不可用或已有仓库基线时 GPT-5.5 | 页面结构与产品目标已确定，并有真实浏览器验收 |
| 按既有模式完成局部 API、adapter、配置、测试或文档 | Terra medium | 不改变公共契约，不涉及未知根因或不可逆状态 |
| 常规资料综合、方案对比和边界明确的代码审阅 | Terra medium / high | 资料来源和评价维度清楚，结论不是高风险最终决策 |
| 定时状态巡检和自动化门禁 | Luna 收集状态；Terra 做常规 reviewer / tester 判断 | 只有触发架构、安全、数据或发布风险时才交给 Sol |

单位 token 费率下，Terra 比 Sol 低 50%，Luna 比 Sol 低 80%，GPT-5.4 Mini 比 Sol 低约 85%。这是降档的直接收益，但实际任务总成本还受模型输出 tokens、工具结果和返工次数影响；如果低档模型因能力不足反复失败，应及时升级，不以多次重试制造虚假节省。

## 候选模型的能力、速度与消耗

OpenAI 当前把 Sol 定位为旗舰复杂任务模型，Terra 定位为日常均衡模型，Luna 定位为清晰、可重复和高吞吐任务模型；GPT-5.5 是上一代复杂工作模型，GPT-5.4 / GPT-5.4 Mini 仍可用于日常与轻量编码。官方 Codex 模型页给出的能力 / 速度是相对档位，不是固定延迟或 tokens/s 保证：

| 模型 | 官方相对能力 | 官方相对速度 | Codex 单位 token 消耗（Luna = 1） | 推荐定位 |
| --- | --- | --- | --- | --- |
| GPT-5.6 Sol（`gpt-5.6-sol`） | 5 / 5 | 2 / 5 | 5× | 复杂、开放、高价值、需要深度判断和最终打磨 |
| GPT-5.5（`gpt-5.5`） | 4 / 5 | 3 / 5 | 5× | 上一代复杂工作模型；用于 Codex cloud、既有验证基线或 5.6 不可用场景 |
| GPT-5.6 Terra（`gpt-5.6-terra`） | 4 / 5 | 3 / 5 | 2.5× | 日常开发、工具协作、既有模式内的工程实现；新任务的默认日常档 |
| GPT-5.4（`gpt-5.4`） | 3 / 5 | 3 / 5 | 2.5× | 5.6 不可用、需要兼容或复现既有行为时的日常工作档 |
| GPT-5.6 Luna（`gpt-5.6-luna`） | 3 / 5 | 4 / 5 | 1× | 明确、重复、低风险、强验收和高吞吐工作 |
| GPT-5.4 Mini（`gpt-5.4-mini`） | 2 / 5 | 4 / 5 | 0.75× | 最便宜的纯机械编码、固定转换、烟测和窄子任务 |

当前公开 Codex credits 费率如下，单位为每 1M tokens：

| 模型 | 输入 | 缓存输入 | 输出 |
| --- | ---: | ---: | ---: |
| GPT-5.6 Sol | 125 credits | 12.5 credits | 750 credits |
| GPT-5.6 Terra | 62.5 credits | 6.25 credits | 375 credits |
| GPT-5.6 Luna | 25 credits | 2.5 credits | 150 credits |
| GPT-5.5 | 125 credits | 12.5 credits | 750 credits |
| GPT-5.4 | 62.5 credits | 6.25 credits | 375 credits |
| GPT-5.4 Mini | 18.75 credits | 1.875 credits | 113 credits |

任务实际总消耗仍取决于上下文长度、reasoning、输出量、工具结果、重试次数和是否启用 fast / ultra；不能只看单 token 价格。对大量纯机械任务，GPT-5.4 Mini 成本最低；需要更强 agent / 工具能力的结构化子任务优先 Luna；对需要多次返工的任务，错误选用低档模型可能比一次使用 Terra 或 Sol 更贵。

横向比较得到以下默认结论：

- **Sol 对 GPT-5.5**：单位 token 同价；Sol 官方能力更高但相对速度更慢。开放式高难任务优先 Sol；GPT-5.5 保留给 Codex cloud、既有工作流基线、延迟敏感且已实测可靠的复杂任务，或 5.6 不可用场景。
- **Terra 对 GPT-5.4**：单位 token 同价且官方相对速度相同，Terra 官方能力更高。新日常开发默认 Terra；GPT-5.4 只在可用性、兼容性、复现旧行为或代表性评测证明它更合适时使用。
- **Luna 对 GPT-5.4**：官方相对能力相同，Luna 更快且单位 token 费率低 60%。明确、重复和高吞吐任务默认 Luna，不再默认 GPT-5.4。
- **GPT-5.4 Mini 对 Luna**：两者官方相对速度相同，Mini 单位 token 费率低约 25%，但能力低一档。纯机械、单命令和固定转换先用 Mini；需要稳定工具协作、多步执行或更强指令遵循时用 Luna。
- 旧模型不是禁止项。仓库对某类任务已有真实成功率、延迟和总成本数据时，以代表性评测覆盖上述默认路由；不要只因为模型更新就丢掉已验证的可靠路径。

公开评测提供以下方向性证据：

- OpenAI 报告 Sol 在 Terminal-Bench 2.1 的命令行 agent 工作流上达到当时最佳结果，并在 GeneBench v1 以少于 GPT-5.5 的 tokens 获得更强结果；这支持把长链路、开放式和工具协调任务交给 Sol，但不能推导为所有任务都更优。
- OpenAI 将 Terra 描述为性能可与 GPT-5.5 竞争但成本减半的日常模型；系统卡中的部分调试与研究任务也显示 Sol / Terra 相对旧模型有提升。
- 系统卡在 ExploitGym 这一特定安全评测上观察到 Luna 大致落在 GPT-5.4 档、Terra 大致落在 GPT-5.5 档、Sol 位于性能 / 输出 token 前沿。该结果只作为相对能力证据，不能直接外推到所有开发任务。
- 系统卡指出，更大的模型在复杂任务中避免编辑冲突的表现通常好于 Terra / Luna，因此复杂合并、重叠写入和不可逆操作不能只因为步骤看起来机械就降级。
- METR 对 Sol 的独立预部署评测认为长任务 time-horizon 估计高度不确定，并观察到作弊与隐藏行为；因此即使使用 Sol，也必须保留明确授权边界、外部验证和主线程审阅，不能把模型档位当成正确性保证。

目前没有足够统一、独立且同一运行环境下的公开数据，可以把三者写成稳定的首 token 延迟或 tokens/s 数值。规则只采用官方相对速度档位；不要拿特定预览、Cerebras 专用通道或单个用户体验数字当作本仓通用速度保证。

## 仓库历史与微基准证据

历史会话分析表明，至少 34.5% 的有效根任务本身就是固定命令、探针、启动、状态或烟测等 Mini / Luna 候选；至少 85.0% 的子任务已经被收窄到 Luna / Terra 可以先承担的范围。实际降档机会主要来自“Sol / Terra 定边界，再把实现、验证和交付拆下去”，而不是让一个便宜模型独立处理完整复杂需求。

2026-07-10 的首次六模型组合型只读微基准中，六个模型都在五分钟内完成；GPT-5.4、Luna、Sol 找到同一条真实并发竞态，GPT-5.4 Mini、GPT-5.5、Terra 未找到。由于它仍属于一次组合审阅，只能作为初步样本。

随后补充了 36 个干净独立会话：六个模型以 medium 分别完成强约束数据处理、并发 bug 修复、前端交互实现和本地 Git 交付，再为每个模型选择代表任务补跑 low / high。所有会话都在五分钟内完成并通过主线程独立验收；Luna 的四任务平均耗时最短、总 credits 最低，Mini 只在最机械的数据任务上取得略低单次 credits。18 个 low / medium / high 样本全部通过，high 没有增加通过率，却让所有模型比 medium 更慢。综合两轮结果得到以下规则：

- Mini 只接真正机械、短上下文、固定步骤的任务，多文件工具协作从 Luna 起步。
- Terra 负责常规工程实现，但关键 correctness 结论需要测试或第二审阅。
- GPT-5.4 / GPT-5.5 只有在兼容性、Codex cloud 或代表性实测优势成立时使用。
- Sol 用于能力上限和关键判断，不因单次高分而覆盖所有阶段。
- 多步受约束开发默认从 medium reasoning 起步；low 只用于重复评测已经证明稳定的机械任务，high 必须由质量收益或风险门槛触发。
- 单次样本不能代表稳定成功率；仓库自己的重复评测和真实验收优先于型号推断。
- 规则分支的独立验收还发现：当 `spawn_agent` 不支持 model / reasoning 时，计划为 Terra / Luna 的两个只读子任务实际都继承了 Sol / xhigh；主会话也因等待超过五分钟 deadline。该结果直接支持“实际档位可核验”和“deadline 必须主动中断”两条硬约束。
- 第二次独立验收成功把 worker 实际隔离为 Terra / medium 与 Luna / medium，但默认 Sol / xhigh 协调器因元数据核验、归档和最终生成耗时 345.7 秒，并把自己的时长误报为约 196 秒。该结果说明只给 worker 降档不够，协调器本身和集成预留也必须进入路由与外部计时。

完整样本、评分、tokens、估算 credits、历史分类和维护方法见 [`model-routing-analysis.md`](./model-routing-analysis.md)。

## Reasoning 的选择

- GPT-5.4 Mini：纯机械短任务从 low 开始；需要多步判断或 high 才能完成时直接升级 Luna / Terra。
- Luna：明确短任务从 low 开始，需要少量多步检查时用 medium；如果任务需要 high / xhigh 才能勉强完成，优先判断是否应升级到 Terra。
- Terra：日常工程任务以 medium 为起点；涉及多文件分析、视觉回归或较多权衡时用 high。边界开始变得开放或高风险时升级 Sol，而不是无限提高 reasoning。
- Sol：复杂任务以 medium / high 起步；只有测得质量收益时使用 xhigh，最困难且质量优先的任务才使用 max。Ultra 只用于能拆成有独立验收面的并行子任务，不能用于上下文强耦合的小改动。
- GPT-5.4 / GPT-5.5：作为已验证或兼容工作流使用时，沿用该工作流已经测过的 reasoning；没有基线时分别按 Terra / Sol 的起点做对照测试，不默认拉到最高。
- 始终使用能通过验收的最低 reasoning。更高 reasoning 通常增加等待时间和 token 消耗，不替代清晰 prompt、正确拆分和真实验证。

## 按任务阶段拆分模型

复杂任务不要绑定一个模型跑到底。先拆成以下阶段，再逐段路由：

| 阶段 | Luna | Terra | Sol |
| --- | --- | --- | --- |
| 探索证据 | GPT-5.4 Mini / Luna 按已知位置检索、收集日志、运行固定复现 | 从多处材料整理事实、识别已有模式 | 在证据不完整时提出假设、决定还需验证什么 |
| 做出决策 | 只做已定义规则下的分类 | 在既有架构内选择实现方式 | 澄清需求、定架构 / 契约、分析未知根因和风险 |
| 实现修改 | GPT-5.4 Mini / Luna 做机械替换、格式、固定模板和窄范围改动 | 边界明确的多文件实现、页面和常规测试 | 核心算法、跨模块契约、高耦合或高风险实现 |
| 验证结果 | GPT-5.4 Mini / Luna 执行已知命令并结构化汇总 | 解释测试 / 浏览器结果并修复局部问题 | 判断异常证据、覆盖盲区和最终高风险验收 |
| 交付操作 | GPT-5.4 Mini / Luna 做 commit、push、PR 元数据、无冲突 merge | 需要理解上下文的 PR 描述和常规审阅 | 冲突解决、历史重写、发布风险和可合入性决策 |

主线程负责阶段边界、授权和最终结果。低档模型只接收已经被上游决策收窄的工作；它发现边界不成立、证据矛盾或需要扩大范围时必须停止并回报，不能自行把任务升级成开放式修改。

## 常见任务类型与例子

下面先给抽象类型，再给例子。例子只是帮助识别，不覆盖所有情况。

### 受约束执行：GPT-5.4 Mini / Luna

特征是“怎么做”和“怎么算完成”都已知，失败可逆且能立即验证。

- 例子：读取仓库状态、定向查找引用、运行已有测试、启动统一 dev-start 命令、整理日志、格式化、按固定模板更新文档或 fixtures。单命令和纯机械转换先用 GPT-5.4 Mini，多步工具链用 Luna。
- Git 例子：在用户已授权、diff 已审阅后生成 commit message，执行 commit / push，创建或更新 PR；required checks 和审批均满足时执行无冲突 merge。流程短且状态清楚时可用 GPT-5.4 Mini，否则用 Luna。
- 不包括：判断代码是否正确、决定是否应该合入、处理冲突、rebase / rewrite 或 force push。

### 日常工程判断：Terra

特征是需要理解若干上下文和使用工具，但工作仍在既有模式、明确边界和验收标准内。

- 例子：根据既有组件和设计系统调整前端页面、样式、响应式与局部交互；按明确契约实现局部 API / 测试；常规 PR 审阅；多来源资料的结构化综合。
- 前端任务默认从 Terra 开始，因为视觉、focus、主题和真实浏览器回归需要判断；纯 token / class 的机械替换或固定回归命令可拆给 Luna。
- 父任务已经给出指定文件、禁止范围、接口契约和验证命令的实现 / 审计子任务默认从 Terra 开始；不要因为父任务使用 Sol 就让所有子任务继承 Sol。
- 如果前端调整暴露复杂状态机、公共组件架构、服务端协议或未知性能根因，升级 Sol。

### 开放式关键判断：Sol

特征是正确路径未知、错误代价高、需要跨边界理解，或验收不能完全自动化。

- 例子：从零开发需求、分析未知 bug 根因、跨前后端 / 跨 package 设计、迁移公共契约、疑难 CI / runtime 问题、安全与权限审阅、数据库迁移、复杂重构。
- Sol 负责需求澄清、方案、核心契约、高风险实现和关键最终审阅；一旦边界固定，应把页面实现、测试补充、日志整理和 Git 流程下放给 Terra / Luna。

## 组合路由示例

### 修复问题

1. Luna 收集日志、定位相关文件并运行已有复现，不判断最终根因。
2. Sol 在证据不完整时建立并验证根因，决定修复和回归范围；如果根因已知且修改模式明确，可由 Terra 直接实现。
3. Terra 按确定方案完成边界清晰的实现和回归测试，Luna 运行固定验证。
4. Sol 审阅高风险或跨模块 diff；通过后由 Luna 完成标准 commit / PR / 无冲突 merge 流程。

### 开发需求

1. Sol 澄清需求，定义契约、模块边界、兼容性和验收标准。
2. Sol 负责核心或高耦合实现；Terra 负责边界确定后的日常工程实现，尤其是前端页面和局部 API。
3. Luna 处理机械改动、fixtures、固定验证和结果汇总。
4. Sol 做跨模块集成与关键审阅；Luna / Terra 完成交付流程。

### PR 与发布流程

1. 代码审阅按变更本身选择：机械小 diff 用 Terra，功能、修复、契约、安全和数据变更用 Sol。
2. 审阅通过后，Luna 执行 commit、push、PR 元数据和检查状态收集。
3. 无冲突、检查与审批满足且用户已授权的 merge 用 Luna；冲突、历史重写、检查异常或发布风险决策交给 Sol。

## 升级、降级与停止条件

- GPT-5.4 Mini 连续失败、需要多步工具判断或遗漏约束时升级 Luna；Luna 需要解释矛盾证据或扩大范围时升级 Terra；出现高风险或开放式决策时直接升级 Sol。
- Terra 发现未知根因、公共契约变化、跨模块冲突或结果无法客观验证时升级 Sol。
- Sol 完成关键决策并把边界固定后，后续机械实现、验证和交付应降级给 Terra / Luna；不要因为任务最初复杂就让所有阶段持续使用 Sol。
- 升级前先区分模型能力不足与 prompt、环境、依赖、工具或任务拆分问题。不要让低档模型无上限重试，也不要用更强模型掩盖错误环境。
- 任何模型遇到缺少授权、真实凭据、破坏性操作、用户选择会改变结果或验收条件无法满足时都必须停止并回到主线程。
