# Medium 编码交付质量

本文规定如何让 medium reasoning 承担大部分日常编码，同时保持全局一致性、合适抽象和可验证交付。`medium` 只是实现阶段的默认推理强度，不是质量证明；质量来自编码前契约、独立审阅、自动化门禁和风险升级。

## 适用范围

- 纯格式、固定替换、单一配置值或完全由确定性检查覆盖的机械修改，可以简化下面的记录，但仍需确认范围和验证命令。
- 任何包含业务逻辑、状态变化、公共接口、多文件协作或用户流程的代码修改，都必须执行本文门禁。
- 修改公共契约、状态所有权、权限、安全、数据、并发、事务、迁移或不可逆状态时，不得只依赖 medium 实现和自审；按风险升级设计或最终审阅。

## 编码前：Change Brief

非机械代码修改在写入前必须先给出简短 Change Brief。可以写在任务计划、子任务 prompt 或 PR 说明中，但至少包含：

- **问题与非目标**：解决什么，明确不做什么。
- **当前责任边界**：负责该行为、数据和生命周期的现有模块；最近的 `AGENTS.md` 和规则入口。
- **影响地图**：入口、调用者、下游依赖、公共类型 / API、配置、持久化、UI 状态和文档是否受影响。
- **必须保持的不变量**：兼容性、错误语义、权限边界、幂等性、缓存 / 重试 / 并发要求。
- **验证与回滚**：局部测试、跨模块检查、真实运行时验证，以及失败时如何恢复。
- **抽象决策**：复用、扩展、新建或保持内联，并说明证据。

如果影响地图或责任边界无法确认，先继续只读取证或升级设计判断，不得靠提高实现者 reasoning 后直接试写。

## 抽象决策门禁

新增共享层、helper、service、hook、adapter、公共组件或通用类型前，必须回答：

1. 它解决的是已经存在的重复 / 稳定变化，还是仅预测未来？
2. 新位置是否真正拥有相关状态、生命周期和错误语义？
3. 它是否减少净概念和耦合，而不是把代码搬进 `utils` 或增加一层转发？
4. 不同调用方的差异是否确实属于同一契约，而不是被参数和条件分支掩盖？
5. 公共 API 是否小于内部实现，并能被独立测试和局部回滚？
6. 复用或小范围扩展现有抽象是否更简单？保持内联是否已经足够？

记录使用以下最小格式：

```text
抽象决策：复用 / 扩展 / 新建 / 保持内联
证据：当前调用方、重复或稳定变化
新增概念：
消除的重复或耦合：
替代方案及未采用原因：
```

证据不足时默认保持局部实现，不做投机性通用化。重复代码本身不是充分理由；只有责任、生命周期和变化方向一致时才提升抽象。

## 实现阶段

- 实现者严格遵守 Change Brief 的允许 / 禁止范围；发现公共契约、未知根因或状态所有权与预期不一致时停止并回报，不自行扩大任务。
- 先复用最近模块的既有模式和入口；新建跨模块路径前重新执行抽象决策门禁。
- 修改行为时同步补充能证明行为的测试；无法自动化的 UI、外部进程或运行时结果必须给出真实验证证据。
- 实现交付必须列出实际改动、与 Change Brief 的偏差、验证结果和未覆盖风险，不能只报告“已完成”。

## 独立双层审阅

实现者不能给自己的产出最终批准。非机械代码修改至少需要一次独立审阅；中高风险或跨模块修改应拆成两个关注面：

### 局部正确性审阅

- 检查逻辑、输入边界、错误路径、资源清理、并发 / 重试、测试真实性和回归范围。
- 必须直接读取 diff 和相关实现，不以实现者总结代替证据。

### 全局与抽象审阅

- 检查修改是否仍属于正确模块，是否遗漏调用方、配置、类型、迁移、文档或用户流程。
- 重新判断“复用 / 扩展 / 新建 / 内联”是否合理，是否产生循环依赖、双重状态源、过度抽象或隐性兼容性变化。
- 主动寻找更简单方案，并检查局部优化是否把复杂度转移给其它模块。

两种审阅可由相同型号和 medium reasoning 的不同干净上下文承担，但不得复用实现者的结论作为批准依据。低风险窄改可以由一个独立 reviewer 同时覆盖两个关注面；若当前工具无法创建独立上下文，主线程必须按两套清单重新读取 diff，并明确说明独立性限制。

## 自动化交付门禁

按风险选择验证，不能用 reasoning 强度替代：

| 变更形态                      | 最低门禁                                    |
| ----------------------------- | ------------------------------------------- |
| 机械窄改                      | 格式、lint、定向测试或可重复命令            |
| 局部业务逻辑                  | 类型检查、相关单测、错误 / 边界用例         |
| 公共类型、API、配置或 adapter | 契约测试、调用方检查、兼容性验证            |
| 跨前后端用户流程              | 相关单测、集成 / 浏览器回归、真实运行时证据 |
| 并发、缓存、重试、事务或迁移  | 竞态 / 幂等 / 失败恢复测试，以及回滚方案    |
| 权限、安全、数据或发布风险    | 独立高风险审阅、完整 CI 和明确授权          |

如果仓库已有架构依赖检查、schema diff、迁移验证、视觉回归或公共 API 检查，必须运行对应入口；没有自动门禁时，把缺口列为剩余风险，不得用 reviewer 口头判断伪装为已验证。

## 升级条件

遇到以下任一情况，停止 medium-only 路径并升级设计或最终审阅；边界确定后，日常实现仍可回到 medium：

- 需求或正确行为存在多种合理解释。
- 根因未知，证据互相矛盾或必须扩大读取 / 修改范围。
- 修改公共契约、跨模块依赖方向或状态所有权。
- 涉及权限、安全、持久化数据、数据库迁移、不可逆远端状态。
- 涉及复杂并发、事务、缓存一致性、性能根因或故障恢复。
- 自动化无法覆盖关键结果，且错误代价较高。
- 两次独立审阅结论冲突，或实现连续遗漏 Change Brief 中的硬约束。

## 交付完成条件

只有以下各项均有证据时才能报告完成：Change Brief 已满足或偏差已解释；抽象决策经过审阅；局部正确性和全局影响已检查；相应自动化 / 真实运行时验证通过；剩余风险和后续项已列出；高风险升级和用户授权要求已满足。
