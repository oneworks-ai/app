# 模型路由持续评测方法

本文规定以后引入新模型、新 reasoning 档位或新工具通道时，如何设计测试、保存证据、分析效果并更新 [`model-routing.md`](./model-routing.md) 的建议工作范围。目标不是得到一个“总榜第一”，而是确认每个模型在哪类任务上达到最低充分能力，以及何时必须升级。

## 触发条件

出现以下任一情况时，重跑相关任务族，不凭版本号或单次体验直接修改建议：

- 新模型、新 preview / stable 版本或新 reasoning 档位可用。
- 官方能力、速度、费率、上下文或工具支持发生变化。
- 委派 / 线程工具开始或停止支持显式 model、reasoning、interrupt 或运行元数据。
- 某任务族连续两次出现超时、越界、返工或关键遗漏。
- 仓库任务分布、架构、主要工具链或验收方式发生明显变化。
- 已有模型退役、不可用，或需要评估兼容性 / 替代路径。

## 评测原则

- **按任务族比较，不按版本推断。** 同一模型可能适合 Git 交付但不适合未知根因；不得用总平均掩盖某个任务族的失败。
- **正确性优先于速度和 credits。** 测试 / 契约硬门禁未过、越界或遗漏关键风险时，该次运行直接失败。
- **比较完整任务成本。** 统计协调器、worker、工具结果、重试、复核和返工，不只比较单位 token 价格或单个 worker。
- **计划值与实际值分开。** model、reasoning、开始 / 完成时间和中断结果以平台 / 外部记录为准；无法核验时写 `unknown`，该样本不能证明实际降档。
- **单次成功只证明可行，不证明稳定。** 更新默认工作范围前必须重复运行并与当前可靠基线同场比较。
- **评测不替代真实交付门禁。** 推荐模型的实际代码仍按 [`code-delivery-quality.md`](./code-delivery-quality.md) 接受全局、抽象、独立审阅和自动验证。

## 固定任务族

代表集至少覆盖以下抽象能力；具体 case 可以变化，但评价维度保持稳定：

| 任务族          | 主要检验                               | 典型验收                                     |
| --------------- | -------------------------------------- | -------------------------------------------- |
| 机械受约束执行  | 指令遵循、固定转换、范围纪律           | 精确输出、格式 / schema、单一命令            |
| 工具与 Git 交付 | 多步工具可靠性、状态判断、授权边界     | 分支 / commit 历史、PR 元数据、无冲突流程    |
| 局部缺陷修复    | 已知范围内的根因理解和回归             | 隐藏测试、错误 / 边界 / 并发用例             |
| 既有模式实现    | 多文件实现、测试同步、不过度扩展       | 行为测试、diff 范围、既有架构一致性          |
| 前端交互与视觉  | UI 模式、状态、focus、主题和浏览器证据 | 真实浏览器、可访问性、视觉 / 交互清单        |
| 局部审阅与诊断  | 找到非显式缺陷、区分事实与风险         | 预埋问题、独立源码核对、误报 / 漏报          |
| 全局与抽象判断  | 责任边界、调用链、契约和抽象质量       | 影响地图、替代方案、独立架构 rubric          |
| 协调与集成      | 拆分、模型路由、等待、汇总和最终预留   | 实际 worker 档位、范围互斥、总时限和完整成本 |

候选模型若只准备承担部分工作，不必跑不相关的高风险实现，但必须跑目标工作范围和相邻升级边界。例如评估机械执行模型时，也要包含一个会触发“停止并升级”的多步 case。

## Case 与环境控制

可自动验收的实现 case 优先复用仓库 [`BENCHMARK.md`](../BENCHMARK.md) 约定：`rfc.md` 是唯一任务输入，候选 patch 与 `patch.test.diff` 在干净基线上验证，测试结果是硬约束，judge 只补充等价实现与关键目标判断。

每次横向比较必须固定：

- 同一任务正文、允许 / 禁止范围、输出格式和停止条件。
- 同一 `base_commit`、依赖、配置、权限、工具版本和机器类别。
- 独立可写 case / worktree，不能让模型互相看到候选结果或共享未提交修改。
- 同一外部 judge、隐藏测试和评分 rubric；测试优先验证行为，不绑定参考实现文本。
- 相同联网 / 安装策略。默认禁止联网和新增依赖，除非联网本身就是任务能力。
- 运行顺序随机或轮换，并记录冷 / 热缓存状态；至少重放一个当前基线模型，避免把平台当天波动误判为模型提升。

## 测试矩阵与重复次数

1. **横向基线**：所有候选先用 `medium` 跑全部目标任务族；每个组合至少 3 次独立运行。
2. **Reasoning 纵向**：只在候选可能承担的代表任务上比较 low / medium / high；每个组合至少 3 次。只有发现稳定质量收益才继续测试更高档。
3. **边界 case**：每个推荐工作范围至少有一个应成功 case 和一个应停止 / 升级 case，检查模型是否知道能力边界。
4. **高风险或高方差**：关键判断、视觉、未知根因和结果波动较大的组合至少 5 次，或明确标记为低置信度。
5. **端到端验收**：微基准通过后，至少运行一次真实分支协调场景；要更新默认路由时建议重复 3 次，覆盖协调器、worker、集成和最终输出。

一次运行可以作为预筛选，但不能单独改变默认建议。预算不足时应减少候选或任务范围，不能用一轮全通过替代重复性。

## 单次运行协议

1. 记录 case 版本、base commit、预期 model / reasoning、运行工具和五分钟内的 deadline。
2. 创建干净隔离环境，并在外部记录实际 `startedAt`。
3. 运行时不向模型泄露参考实现、隐藏测试、其它模型答案或期望排名。
4. 4 分 30 秒要求停止探索并交付已有结果；5 分钟由外部监控主动中断。超过即记 `timeout` / 不能用于该任务族，不接受模型自报时间覆盖。
5. 提取 patch / 报告后，在干净基线上执行硬测试，再由独立 judge 检查任务完成度、范围和非自动化质量。
6. 记录实际 model / reasoning、平台时长、tokens、缓存输入、估算 credits、重试和返工。
7. 结果提取后归档会话；归档和统计在任务 final reserve 之外执行。

## 原始数据字段

原始结果保存到本地忽略目录或专用结果存储，至少包含：

```text
evaluation_version, run_id, task_family, case_id, case_version
base_commit, environment_version, cache_state
planned_model, planned_reasoning, actual_model, actual_reasoning
started_at, completed_at, duration_ms, timeout_ms, status
input_tokens, cached_input_tokens, output_tokens, total_tokens, estimated_credits
test_exit_code, hard_gate_pass, judge_score, critical_miss
scope_violation, escalation_correct, retry_count, rework_count
review_findings, archive_status
```

公开或提交到仓库的报告只保留聚合数据和可复现方法，不写本地绝对路径、真实会话 ID、账号、secret、个人内容或仓库无关历史。对外分享包使用通用临时项目和脱敏 case。

## 分析方法

按任务族分别输出通过率、关键遗漏率、越界率、正确升级率、P50 / P95 时长、平均 / P95 credits、tokens、重试和返工。分析时：

- 先排除环境 `error`；模型 `failed`、`timeout` 和错误扩展范围必须保留在分母中。
- 把硬测试、独立 judge 和真实运行时证据分开，不用一个主观总分覆盖硬失败。
- 报告中同时列当前推荐基线和候选，标出样本数、日期、环境和置信度。
- 关注总任务成本与尾延迟；便宜但经常重试的模型不能按单次成功成本获胜。
- 检查质量是否来自更高 reasoning；若 low / medium 已同样通过，不推荐 high。
- 对架构与抽象任务记录 reviewer 找到的遗漏和返工，而不只记录最终测试是否通过。
- 对协调场景拆出 coordinator 和每个 worker 的实际成本，防止父线程隐藏总体浪费。

## 更新推荐工作范围

只有候选在目标任务族满足以下条件时，才扩大默认范围：

- 没有关键行为失败、越权、不可接受的 timeout 或错误风险决策。
- 重复运行的通过率不低于当前可靠基线，P95 时长满足任务 deadline。
- 总 credits / 延迟有明确收益，或质量提升足以解释更高成本。
- 相邻边界 case 能正确停止并升级，不会把开放 / 高风险任务误当作机械执行。
- 真实分支验收中实际 model / reasoning 可核验，协调器和 worker 的总成本仍成立。
- 独立 reviewer 确认结论没有从单一 case 过度外推。

如果候选只在某类任务更好，只更新该任务族、前提和升级条件；不要把局部胜出写成全局替代。连续回归、关键遗漏或工具兼容性下降时收窄 / 暂停推荐，并保留最近可靠模型作为回退。

更新文档的职责固定为：

1. [`model-routing-analysis.md`](./model-routing-analysis.md) 记录日期、样本、结果、失败和置信度。
2. [`model-routing-models.md`](./model-routing-models.md) 更新当前官方能力、速度、费率与可用性基线。
3. [`model-routing.md`](./model-routing.md) 只更新经证据支持的默认工作范围、reasoning 和升级边界。
4. 根 `AGENTS.md` / 默认维护入口只在结论稳定且跨任务适用时更新硬约束，避免把临时榜单写进默认 prompt。
5. 修改后重新运行独立分支验收、文档检查和独立经验 reviewer；PASS 与 CI 全绿后才能合入。
