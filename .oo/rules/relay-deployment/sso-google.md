# Relay Google SSO

Read `.oo/rules/relay-deployment/sso-common.md` first.

## Recommended Shape

- The Google OAuth client must be a Web application client.
- Use a product-facing OAuth consent/app name such as `OneWorks Login` or the user's own brand.
- Keep dev and production clients separate when the project, audience, consent state, or callback set should be isolated.
- Do not store Google client secrets in committed files; use platform secret storage or local ignored scratch files.

## Callback Setup

Callback paths:

```text
<public-origin>/api/auth/oauth/google/callback
<public-origin>/api/auth/oauth/<managed-google-provider-id>/callback
```

- The built-in Google provider uses `/api/auth/oauth/google/callback`.
- A Google provider created from Admin uses the configured provider id, for example `/api/auth/oauth/google-sso/callback`.
- The redirect URI registered in Google Cloud must exactly match the URL Relay sends, including scheme, host, path, and port.
- In split deployments, register the public Admin/API/login origin, not a hidden Worker or function URL.

## Relay Identity Behavior

- Google / generic OIDC login must require `email_verified === true`.
- Relay should treat Google `sub` as the provider user id.
- Relay identities are keyed by provider id and Google subject, not by email.
- The same verified email from Google and GitHub should create separate Relay users.

## Troubleshooting

- `redirect_uri_mismatch` means the Google callback list and Relay's effective public origin are out of sync. Check `ONEWORKS_RELAY_PUBLIC_URL`, any Admin proxy origin, and the exact provider id path.
- Google compares redirect URIs byte-for-byte. `localhost` and `127.0.0.1` are different; `http` and `https` are different; every port and path segment must match, including `/api/auth/oauth/<provider-id>/callback`.
- Local Relay demos often use one port for the Admin/login page and another for a helper/mock service. Register the callback origin actually generated by Relay, not the helper service origin.
- If local validation needs multiple loopback ports, either add every exact callback to the dev OAuth client or pin `ONEWORKS_RELAY_PUBLIC_URL` to an origin that is already registered.
- Keep Google dev and production OAuth clients separate when consent screen state, callback lists, or secret exposure should not be shared.
- If Google login returns an email without `email_verified=true`, reject it instead of binding accounts by email.
- If the login page shows the wrong icon, update provider icon mapping and the login provider icon component/tests.
